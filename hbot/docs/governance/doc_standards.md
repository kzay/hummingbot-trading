# Documentation Standards

## Purpose
Keep project documentation consistent, reviewable, and operationally useful.

## Required Sections (for spec docs)
- Purpose
- Scope
- Assumptions
- Inputs/Outputs
- Failure Modes
- Source of Truth references
- Owner
- Last-updated

## Style Rules
- Keep thresholds numeric and traceable to code/config paths.
- Avoid duplicate policy text; reference canonical docs.
- Use concise diagrams for architecture/data-flow topics.

## Review Policy
- Any feature PR touching behavior should update impacted docs.
- Reviewer must confirm links and source paths are valid.

## Ownership
- Primary: Engineering
- Secondary: Risk/Ops for policy and runbook sections
